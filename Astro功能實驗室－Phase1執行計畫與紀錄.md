# Astro 功能實驗室 Phase 1・執行計畫與紀錄

- 建立日期：2026-07-15
- 目前狀態：Step 1～Step 4 已完成；下一步為 Step 5
- 前置狀態：Phase 0・Step 1～Step 6 已完成並由使用者驗收
- 依據文件：`厄倫蒂兒粉絲應援網站－開發計畫書.md`、`Astro功能實驗室－企劃書.md`、`Astro功能實驗室－Phase0-Step6-QA.md`

## 1. 文件目的

本文件是 Phase 1 的主控文件，同時用於：

- 規劃 Phase 1A「架構基線」與 Phase 1B「原生內容體驗」的執行順序。
- 限制每次只處理一個 Step，避免多個功能一起開發而無法個別驗收。
- 記錄每個功能的實作範圍、測試證據、人工驗收與採用決策。
- 在加入任何套件前，先完成原生方案與 Astro 內建能力的比較。
- 讓每個 Step 即使判定「延後」或「不採用」，仍能留下完整結論後關閉。

本文件只規劃與追蹤 Phase 1，不代表已核准開始實作。每個 Step 開始前仍需由使用者確認範圍；完成技術驗證後，也必須由使用者實際查看與驗收，才能進入下一個 Step。

## 2. Phase 1 範圍與完成定義

### 2.1 Phase 1A：架構基線

包含：

- F18 Content Collections／動態活動頁。
- F20 圖片最佳化與 Lazy Loading。
- F26 動態偏好與效能模式的共用介面與整合驗證。
- F03 Navigation。
- F23 多語系路由決策。
- F04 Page Transition。
- Phase 1A 整合 QA 與架構凍結。

完成 Phase 1A 時，必須明確固定或記錄下列決策：

- 活動資料 schema 與錯誤資料的 build-time 驗證方式。
- 活動列表、詳情頁與媒體資源的 URL 結構。
- 圖片來源、尺寸、格式、載入優先序與 fallback 策略。
- 所有後續實驗讀取動態、音效與效能偏好的共同方式。
- 單頁錨點、多頁路由、深層連結、手機導覽與返回鍵行為。
- 多語系採用、延後或不採用；不可保持未決狀態。
- 使用普通 MPA、原生 View Transition 或 Astro ClientRouter。
- 頁面切換後 script 的統一初始化與清理規則。

### 2.2 Phase 1B：原生內容體驗

包含：

- F05 Scroll Animation。
- F06 Gallery。
- F07 Carousel。
- F08 Lightbox。
- F09 自有 Video。
- F14 YouTube 影片嵌入。
- F10 Audio Player。
- F17 按鍵音效。
- F22 分享、複製連結與 QR Code。
- F19 搜尋、標籤與篩選。
- Phase 1B 整合 QA 與 Phase 1 結案。

完成 Phase 1B 時，每一項功能都必須有「原生方案足夠」、「核准採用套件」、「延後」或「不採用」的明確結論。不能只因 Prototype 能顯示就視為完成。

### 2.3 不在本階段

下列功能保留至後續 Phase，不得趁 Phase 1 順便加入：

- Phase 2：X、Threads、動畫滑鼠。
- Phase 3：Interactive Islands 框架決策、人物動畫／Live2D、桌寵、Canvas／WebGL、小遊戲。
- Phase 4：投稿表單後端、PWA／Service Worker。
- 正式 CMS、正式上線、正式網域與大量內容移轉。

## 3. 執行與驗收規則

### 3.1 Step 狀態

每個 Step 只能使用下列狀態：

| 狀態 | 定義 |
|---|---|
| 未開始 | 尚未由使用者確認本 Step 的範圍 |
| 執行中 | 範圍已確認，正在實作或驗證 |
| 待驗收 | 技術檢查已完成，等待使用者實際查看或操作 |
| 完成 | 技術檢查與使用者驗收皆完成，決策已記錄 |
| 阻擋 | 有具體外部條件使本 Step 無法繼續，已記錄解除條件 |
| 延後 | 已完成必要研究，但決定不在 Phase 1 實作 |
| 不採用 | 已完成比較並留下理由，正式排除該方案 |

### 3.2 每個 Step 的固定流程

1. 閱讀本 Step 與前置 Step 的決策紀錄。
2. 由使用者確認本輪範圍、素材與可接受的驗收方式。
3. 記錄開始前的 Git 狀態、既有 dependency 與 build 基線。
4. 只實作本 Step 核准的最小 Prototype，不提前做後續功能。
5. 執行 build、靜態檢查、功能測試與必要的回歸測試。
6. 將自動檢查與人工檢查分開記錄，不把未實際執行的項目標為通過。
7. 由使用者在本機頁面實際查看或操作。
8. 記錄問題、修正、最終決策、工時與估算誤差。
9. 使用者明確確認後才將 Step 改為完成並進入下一步。

### 3.3 共通品質底線

每個可見功能至少檢查：

- 360、768、1440 px 的版面與操作。
- 鍵盤操作、焦點樣式、語意與必要的螢幕閱讀器狀態。
- `prefers-reduced-motion` 與 Lab 站內動態偏好。
- JavaScript 失效時，主要內容仍可閱讀或有明確 fallback。
- 載入失敗、資料缺漏、無結果與不支援 API 的狀態。
- 普通進入、重新整理、直接開深層網址、前進與返回。
- `pnpm build:lab` 通過，並視變更範圍執行 `pnpm build` 回歸。
- Lab 的 noindex、正式 build 排除與既有首頁不被意外破壞。
- 沒有未核准 dependency，`git diff --check` 無格式錯誤。

跨瀏覽器、Safari、實體手機與 Lighthouse 若受環境限制，必須標記為「未執行／延後」，不能推定通過。

### 3.4 套件決策閘門

新增 dependency 前必須先記錄：

- 原生或 Astro 內建方案的 Prototype 結果。
- 無法達成的具體驗收條件，而非只以開發方便為理由。
- 候選套件的授權、維護狀態、Astro／Node 相容性與 bundle 成本。
- 頁面切換、重複初始化、事件清理、reduced-motion 與無 JavaScript fallback 的影響。
- 使用者對新增套件的明確核准。

一次只核准一個候選套件。安裝後若未通過驗收，應移除並確認 lockfile、build 與頁面已恢復乾淨。

## 5. 各 Step 執行規格

### Step 1：F18 Content Collections／動態活動頁

**目標**

建立活動、圖片、影音與社群連結的共同資料來源，避免後續 Gallery、影音、分享與搜尋各自使用不同資料格式。

**本 Step 範圍**

- 以 Astro Content Collections 定義活動 schema。
- 準備至少 3 筆合法測試資料，涵蓋不同日期、標籤與媒體組合。
- 建立 Lab 內的活動列表與動態詳情頁。
- 驗證日期排序、slug、標籤、選填欄位與相關資料的基本使用方式。
- 建立錯誤資料測試，確認缺欄位、錯日期或無效型別會在 build 階段失敗。

**不在本 Step**

- 正式視覺、搜尋介面、Carousel、Lightbox、影音播放器與 CMS。

**完成閘門**

- 3 筆合法資料能產生列表與獨立詳情網址。
- 直接開啟詳情網址與重新整理皆正確。
- 錯誤資料確實能被 schema／build 發現，測試後不留在正式資料中。
- schema、URL 與選填欄位決策已記錄並由使用者驗收。

### Step 2：F20 圖片最佳化與 Lazy Loading

**目標**

建立後續所有圖片型功能共用的來源、尺寸、格式、載入優先序與 fallback 策略。

**本 Step 範圍**

- 使用同一批高解析圖片比較原圖與 Astro 圖片能力。
- 驗證 responsive `srcset`／`sizes`、WebP／AVIF、尺寸預留與裁切焦點。
- 區分首屏優先圖片與非首屏 lazy loading。
- 定義本機圖片、遠端圖片、透明圖片與載入失敗的處理方式。
- 記錄檔案大小、輸出尺寸、畫質與 CLS 觀察。

**完成閘門**

- 360、768、1440 px 取得合理圖片尺寸，不全數下載桌機大圖。
- 圖片區塊預留尺寸，無明顯版面跳動。
- 首屏與非首屏載入策略不同且可驗證。
- 失敗圖片有可接受 fallback，圖片策略能供 Step 9～13 共用。

### Step 3：F26 共用偏好介面與整合驗證

**目標**

沿用 Phase 0 已完成的控制器，不重做設定 UI；本 Step 要固定後續功能讀取與回應偏好的共同契約。

**本 Step 範圍**

- 盤點現有 `data-motion`、`data-sound`、`data-performance` 與 localStorage 行為。
- 定義 CSS、Astro 元件與 TypeScript 功能讀取偏好的唯一方式。
- 定義偏好變更時的通知方式，以及頁面切換後的狀態恢復方式。
- 建立最小測試案例，驗證完整動態／減少動態、靜音、標準／節能模式。
- 驗證系統 reduced-motion 與站內設定不衝突，重設可恢復預期值。

**完成閘門**

- 後續功能不需要各自建立第二份 localStorage 或 media query 狀態管理。
- 偏好在重新整理與 Lab 頁面間切換後保持一致。
- JavaScript 或儲存空間不可用時有安全預設。
- 共用契約與使用範例已記錄。

### Step 4：F03 Navigation

**目標**

驗證活動列表、詳情頁與 Lab 子頁之間的導覽，固定深層連結、返回鍵、手機選單與目前位置提示的行為。

**本 Step 範圍**

- 多頁導覽、單頁錨點、活動列表與詳情頁往返。
- 桌機與手機導覽，包含開啟、關閉、Escape 與焦點返回。
- `aria-current`、Skip Link、麵包屑或返回上一層的必要方案。
- 直接開啟 `#section` 與子頁網址。
- JavaScript 關閉時的可用導覽 fallback。

**完成閘門**

- 鍵盤能完整操作且焦點順序合理。
- 手機選單不遮擋、不產生關閉後仍可聚焦的內容。
- 前進、返回、重新整理與直接開網址符合瀏覽器預期。
- Navigation 規則可供多語系與 Page Transition 使用。

### Step 5：F23 多語系路由決策

**目標**

在頁面數量增加前，明確決定正式站的多語系策略；本 Step 的完成結果可以是採用、延後或不採用。

**本 Step 範圍**

- 使用最小繁中／日文／英文 placeholder 驗證候選 URL 結構。
- 比較預設語言是否帶 prefix、語言切換與對應頁 fallback。
- 檢查日期格式、文字長度、字型、換行、`lang` 與 `hreflang` 需求。
- 評估翻譯、校對、缺翻譯與內容維護成本。
- 優先評估 Astro 內建 i18n，不安裝翻譯套件。

**完成閘門**

- 留下「Phase 1 採用」、「正式站前延後」或「不採用」的單一結論。
- 若採用，固定 URL 與內容 fallback；若延後，確保目前 schema 與路由未封死未來擴充。
- 使用者核准決策，不能以「之後再看」結束本 Step。

### Step 6：F04 Page Transition

**目標**

比較普通 MPA、原生 View Transition 與必要時的 Astro ClientRouter，固定後續所有 script 的生命週期規則。

**本 Step 範圍**

- 至少使用列表與詳情兩個獨立頁面測試前進、返回、重整與直接進入。
- 比較無轉場／短淡入與一個共享元素 Prototype。
- 驗證 reduced-motion、未支援瀏覽器與 JavaScript 關閉的降級。
- 若評估 ClientRouter，驗證初始化、事件移除、焦點、捲動位置與重複執行。
- 記錄第三方 script、影音與後續互動功能必須遵守的初始化規則。

**完成閘門**

- 明確決定 MPA、原生 View Transition 或 ClientRouter。
- 前進、返回、重整、深層連結與連續切頁不出錯。
- 沒有重複 listener、重複初始化或返回鍵異常。
- reduced-motion 與不支援環境能正常換頁。

### Step 7：Phase 1A 整合 QA 與架構凍結

**目標**

在開始內容互動功能前，確認資料、圖片、偏好、路由與 script 生命週期已能穩定共存。

**本 Step 範圍**

- 回歸 Step 1～6 的自動、靜態與人工檢查。
- 驗證正式／Lab 雙模式 build、noindex、正式首頁與 404。
- 建立 Phase 1A QA 紀錄，列出通過、未執行、已知限制與延後項目。
- 將 schema、URL、圖片、偏好、i18n 與轉場決策標記為 Phase 1B 的固定前置。

**完成閘門**

- Step 1～6 均為完成、延後或不採用，沒有未決狀態。
- 使用者完成 Phase 1A 人工驗收。
- Phase 1A QA 紀錄可作為 Step 8～17 的共同基線。

### Step 8：F05 Scroll Animation

**目標**

建立可重複使用的原生 scroll reveal 基線，並確認是否真的需要 GSAP／ScrollTrigger。

**本 Step 範圍**

- 以 CSS、Intersection Observer 或可用的 Scroll-driven Animations 實作少量代表案例。
- 驗證內容先存在於 HTML，動畫只作漸進增強。
- 處理重複進入、返回頁面、版面高度改變與 observer 清理。
- 手機降低位移與 sticky 成本；reduced-motion 直接顯示內容。

**完成閘門**

- 內容不必等待動畫才能閱讀。
- 沒有明顯 CLS、手勢阻斷或重複初始化。
- 360 px 與節能模式可接受。
- 得出「原生足夠」或「另開 GSAP 套件核准」結論。

### Step 9：F06 Gallery

**目標**

建立 Gallery 作為 Carousel 與 Lightbox 的共同資料、圖片與 caption 容器。

**本 Step 範圍**

- 使用至少 12 張不同長寬比測試圖片。
- 實作響應式 CSS Grid、圖片說明、alt、作者／授權欄位與載入失敗狀態。
- 驗證重要圖片裁切焦點、Lazy Loading 與鍵盤可達性。
- 判斷是否真的需要 Masonry；本 Step 不預設加入套件。

**完成閘門**

- 桌機、平板與手機欄數和觸控目標合理。
- 圖片載入不造成明顯跳動，失敗時不破壞網格。
- Gallery 資料可直接供 Carousel 與 Lightbox 使用。

### Step 10：F07 Carousel

**目標**

使用 Gallery 資料測試原生 CSS Scroll Snap 的品質上限，再決定是否需要 Embla Carousel。

**本 Step 範圍**

- 6～10 張內容，包含上一張／下一張、分頁、目前位置與觸控滑動。
- 原生 Scroll Snap、鍵盤、滑鼠與觸控操作。
- 手機顯示下一張提示，按鈕與分頁不超出畫面。
- 不預設自動播放；若測試自動播放，必須可暫停。

**完成閘門**

- 鍵鼠、觸控與鍵盤皆可用。
- 焦點、目前頁狀態與內容可讀性正確。
- 得出「原生 Scroll Snap」或「核准評估 Embla」的明確決策。

### Step 11：F08 Lightbox

**目標**

用 Gallery／Carousel 的同一份資料建立原生 Lightbox，驗證 `<dialog>` 是否足夠。

**本 Step 範圍**

- 放大圖片、上一張／下一張、caption、作者資訊與關閉控制。
- Esc、焦點鎖定、關閉後焦點返回、背景不可誤操作。
- 手機安全區、直橫向圖片與必要的滑動／縮放需求評估。
- 直接開啟大圖失敗時的 fallback。

**完成閘門**

- 鍵盤與螢幕閱讀器能理解開啟／關閉狀態。
- 關閉後回到原縮圖，瀏覽位置不遺失。
- 手機控制不被瀏海或系統區域遮擋。
- 得出「原生 `<dialog>`」或「核准評估 PhotoSwipe」的明確決策。

### Step 12：F09 自有 Video

**目標**

建立本站可控制的影片容器、poster、字幕、載入與動態偏好降級規則。

**本 Step 範圍**

- 分別測試內容影片與背景影片的最小案例。
- MP4／WebM、poster、VTT 字幕、原生控制與錯誤 fallback。
- 手機不搶首屏頻寬；背景影片保持靜音。
- reduced-motion 與節能模式顯示 poster 或不自動播放。

**完成閘門**

- 有聲播放只能由使用者觸發。
- 內容影片保留鍵盤可用的控制與字幕。
- 背景影片、內容影片、手機與 reduced-motion 各有明確降級方式。
- 載入失敗時仍能顯示 poster、說明或替代連結。

### Step 13：F14 YouTube 影片嵌入

**目標**

沿用 Step 12 的媒體容器，以點擊後載入 facade 降低首屏成本與不必要的第三方連線。

**本 Step 範圍**

- 使用官方 iframe 或 IFrame Player API，不安裝 wrapper 套件。
- 初始只顯示合法 poster、標題、頻道與播放按鈕。
- 點擊後才載入選定影片，多影片時不一次建立全部 iframe。
- 處理禁止嵌入、刪除、年齡限制、網路失敗與外部連結 fallback。
- 驗證 Page Transition 決策下的初始化與清理。

**完成閘門**

- 16:9、鍵盤、全螢幕與手機操作正確。
- 未點擊前不建立不必要的播放器連線或 iframe。
- 連續切頁或返回不會重複建立播放器。
- 失效影片有可理解的 fallback。

### Step 14：F10 Audio Player

**目標**

建立全站唯一的主要音訊管理方式，作為按鍵音效的共同基礎。

**本 Step 範圍**

- 原生 `<audio>` 的播放／暫停、進度、曲目資訊與必要音量策略。
- 一次只播放一個主要音源。
- 音效預設關閉、使用者手勢啟動與全站靜音偏好。
- 切頁、離開可視區、重新整理與播放失敗的行為決策。
- 波形只做需求評估，不預設加入 WaveSurfer。

**完成閘門**

- 鍵盤可操作，狀態文字清楚。
- 不會自動出聲，不會同時播放多個主要音源。
- 切頁行為符合 Step 6 的架構決策。
- 得出「原生足夠」或「因明確波形需求另開套件核准」結論。

### Step 15：F17 按鍵音效

**目標**

共用 Step 14 的 audio manager，驗證少量按鍵回饋而不為每個按鈕建立獨立 Audio 實例。

**本 Step 範圍**

- 至少成功／取消兩類短音效。
- 只在使用者啟用聲音且已發生互動後播放。
- 快速連點、重複 hover、切頁與節能模式行為。
- 永久關閉與重設；不安裝 Howler。

**完成閘門**

- 初始不出聲，靜音偏好立即生效。
- 快速連點不疊加成過大音量。
- 按鍵沒有聲音時仍有完整視覺或文字回饋。
- 共用 Audio 管理方式，沒有第二套狀態。

### Step 16：F22 分享、複製連結與 QR Code

**目標**

以穩定的活動 URL、標題與摘要建立手機分享和桌機 fallback。

**本 Step 範圍**

- Web Share API 與不支援時的 Clipboard／手動複製 fallback。
- 分享標題、摘要、活動深層連結與成功／失敗提示。
- 驗證返回、直接開啟與含特殊字元 URL。
- QR Code 先決定是否需要；若需要，再比較 build-time 產生或小型套件。
- Open Graph 圖僅定義需求，不在本 Step 完成正式社群平台 QA。

**完成閘門**

- 手機可呼叫系統分享；桌機或不支援環境能複製或手動選取網址。
- 成功、拒絕與錯誤都有可理解提示。
- 分享網址可直接開到正確活動頁。
- QR Code 有採用、延後或不採用決策；新增套件需另行核准。

### Step 17：F19 搜尋、標籤與篩選

**目標**

以穩定活動 schema 建立原生搜尋與篩選基線，並判斷目前資料量是否值得加入 Pagefind。

**本 Step 範圍**

- 年份、標籤或活動類型篩選，以及關鍵字搜尋。
- URL 保留條件，重新整理與返回鍵能恢復狀態。
- 結果數、清除條件與無結果狀態。
- 手機可收合但仍保有鍵盤與觸控可用性。
- JavaScript 關閉時仍顯示完整活動列表。

**完成閘門**

- 鍵盤可操作，URL、重新整理與返回鍵狀態一致。
- 無結果與資料缺少 tag 時不出錯。
- 無 JavaScript 時內容仍可閱讀。
- 得出「原生足夠」、「資料量不足先延後」或「另開 Pagefind 套件核准」結論。

### Step 18：Phase 1B 整合 QA 與 Phase 1 結案

**目標**

驗證 Phase 1A 架構與 Phase 1B 所有採用功能能同時運作，完成 Phase 1 的最終交接紀錄。

**本 Step 範圍**

- 回歸 Step 1～17 的資料、圖片、偏好、導覽、轉場與內容互動。
- 檢查快速連續切頁、重複開關媒體、返回、深層連結與失敗狀態。
- 執行正式／Lab 雙模式 build、noindex、Sitemap、dependency 與首頁／404 回歸。
- 分開記錄自動、靜態、人工與受環境限制未執行的 QA。
- 彙整各功能的原生／套件／延後／不採用決策與實際工時。
- 建立 Phase 1 QA／交接文件，列出 Phase 2 是否可以開始及其前置條件。

**完成閘門**

- Step 1～17 全部有最終狀態與使用者驗收紀錄。
- 沒有未核准 dependency 或未記錄的架構分歧。
- 採用功能在 360、768、1440 px 與鍵盤操作下達到共通品質底線。
- 使用者確認 Phase 1 完成；未完成的跨瀏覽器、實機或 Lighthouse 項目被明確列為延伸 QA。
