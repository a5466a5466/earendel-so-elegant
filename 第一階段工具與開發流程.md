# 厄倫蒂兒粉絲應援網站：第一階段工具與開發流程

版本：v1.0  
日期：2026-07-14  
階段目標：在本機完成可操作、可建置的網站原型；暫不部署、不使用後端、不加入 React。

## 1. 已確認的技術決策

```text
主框架：Astro
專案初始化：pnpm create astro@latest
主要 UI：Astro＋HTML＋原生 CSS
主要互動：原生 TypeScript
內容：Markdown＋Astro Content Collections
套件管理：pnpm（禁止和 npm／Yarn 混用）
版本控制：Git
React：暫不安裝，需要時可在後續階段加入
```

第一階段只在本機開發。Cloudflare Pages、Cloudflare R2、正式網域與部署流程全部延後，不會影響本機網站製作。

## 2. 第一階段的完成定義

第一階段不是完成正式網站，而是完成一個「技術基礎可靠的本機原型」：

- Astro 專案能以 `pnpm dev` 啟動。
- 能以 `pnpm build` 成功產生靜態網站。
- 首頁、活動列表、活動詳情、關於頁與 404 頁可瀏覽。
- 至少放入 3 筆 Markdown 活動資料。
- 已建立基本色彩、字體、間距與 RWD 規則。
- 導覽、手機選單、基本圖片藝廊使用原生 TypeScript。
- 至少有一個原生 CSS Scroll Snap Carousel prototype。
- 簡單入場與 hover 動畫使用 CSS。
- 不依賴 React、Vue、Svelte、GSAP、Embla 或雲端服務。
- Chrome、Edge、Firefox 與手機尺寸測試沒有阻斷問題。

## 3. 必須安裝的電腦工具

### 3.1 Visual Studio Code

用途：編輯 `.astro`、HTML、CSS、TypeScript 與 Markdown。

目前狀態：已安裝（檢查到 VS Code 1.128.0）。

第一階段建議的 VS Code 擴充套件：

| 擴充套件 | 必要性 | 用途 |
|---|---|---|
| Astro（官方） | 必裝 | `.astro` 語法、型別、錯誤提示與自動完成 |
| GitHub Copilot | 已決定使用 | 程式碼建議與對話協作 |
| Prettier | 建議 | 統一格式，降低 AI 產生不同排版的情況 |
| Error Lens | 選配 | 把 TypeScript／Astro 錯誤直接顯示在程式旁 |

不要安裝大量 snippet、主題型 UI builder 或來源不明的 Astro 擴充套件。

### 3.2 Node.js 24 LTS

用途：執行 Astro、pnpm、TypeScript 檢查與建置工具。

目前狀態：一般 PowerShell 的 PATH 中找不到 Node.js，需要安裝。

規則：

- 使用 Node.js 24 LTS，不使用 Current、beta 或 nightly。
- 專案加入 `.nvmrc` 或等效版本紀錄，內容鎖定 major 版本 `24`。
- 安裝後重新開啟 VS Code，再確認：

```powershell
node --version
```

應顯示 `v24.x.x`。

### 3.3 pnpm

用途：安裝 Astro 與其他專案依賴、執行開發指令、產生 `pnpm-lock.yaml`。

目前狀態：Codex bundled runtime 中有 pnpm 11.7.0，但仍需確認一般 VS Code PowerShell 可以在 Node.js 安裝後直接執行：

```powershell
pnpm --version
```

專案規則：

- 只保留 `pnpm-lock.yaml`。
- 不產生 `package-lock.json` 或 `yarn.lock`。
- `package.json` 設定 `packageManager`，鎖定使用中的 pnpm 版本。
- AI 不得自行刪除 lockfile、改用 npm 或一次升級所有依賴。

### 3.4 Git

用途：記錄版本、建立安全還原點、比較 AI 修改內容。

目前狀態：已安裝 Git 2.54.0，工作區也已是 Git repository。

第一階段只需要本機 Git；GitHub remote 可以稍後再設定。

建議提交節點：

1. Astro 初始化完成。
2. 全站 Layout 與設計 token 完成。
3. Content Collections 完成。
4. 每個主要頁型完成。
5. 第一階段 QA 完成。

## 4. Astro 專案內的核心工具

### 4.1 Astro

Astro 負責：

- 頁面與網址路由。
- Layout 和可重用元件。
- 把 Markdown 活動內容轉成頁面。
- 圖片處理與靜態建置。
- 打包原生 TypeScript。
- 開發伺服器與 production build。

第一階段採用 Astro 的 `minimal` 起始模板，不加入 blog template、Tailwind 或任何 UI framework。

互動式初始化時，建議選擇：

```text
Project directory：目前 repository 根目錄或確認後的網站目錄
Template：minimal
Install dependencies：Yes
Initialize Git：No（目前已經有 Git repository）
```

建立後立即提交 `package.json` 與 `pnpm-lock.yaml`，從此不使用 `@latest` 執行日常開發；`@latest` 只用在首次建立專案。

### 4.2 TypeScript strict mode

Astro 元件的 frontmatter 和 `<script>` 都可以使用 TypeScript。

第一階段使用 Astro 的 strict TypeScript 設定，目標是讓：

- `querySelector` 可能找不到元素時必須處理。
- 元件 props 有明確型別。
- 活動資料欄位有明確型別。
- AI 產生錯誤欄位時能在建置前被發現。

禁止為了讓錯誤消失而大量使用 `any`、`@ts-ignore` 或非必要的 `!`。

### 4.3 原生 CSS

第一階段不使用 Tailwind、Bootstrap、UI component library 或 CSS-in-JS。

CSS 分成：

```text
src/styles/
├── reset.css       瀏覽器基礎差異
├── tokens.css      顏色、字體、間距、圓角、陰影、動畫時間
├── global.css      全站排版與共用元素
└── utilities.css   少量真正重複的輔助 class
```

單一元件專屬樣式優先寫在對應的 `.astro` 元件中，利用 Astro scoped CSS 避免互相污染。

### 4.4 原生 TypeScript

第一階段互動全部使用瀏覽器 API：

- `addEventListener`
- `classList`
- `dataset`
- `IntersectionObserver`
- `ResizeObserver`（確實需要時）
- `matchMedia`
- `requestAnimationFrame`
- CSS Scroll Snap

每個功能都應有獨立模組，不把所有程式塞進一個 `main.ts`：

```text
src/scripts/
├── mobile-menu.ts
├── scroll-reveal.ts
├── carousel.ts
├── lightbox.ts
└── custom-cursor.ts
```

第一階段不啟用 Astro `<ClientRouter />`，因此先使用傳統頁面導航，避免 script 重新初始化與第三方元件生命週期問題。

### 4.5 Markdown＋Astro Content Collections

每個活動使用一份 Markdown：

```text
src/content/events/
├── birthday-2025.md
├── anniversary-2026.md
└── fan-project-2026.md
```

由 `src/content.config.ts` 定義 schema，至少驗證：

- `title`
- `summary`
- `date`
- `cover`
- `tags`
- `featured`
- `gallery`
- `videos`
- `audio`
- `socialPosts`

第一階段不需要額外 CMS 或資料庫。

## 5. 程式品質工具

### 5.1 Astro Check

安裝 Astro 官方檢查工具與 TypeScript，提供 `.astro`、props 和內容型別檢查。

預計建立指令：

```json
{
  "scripts": {
    "dev": "astro dev",
    "check": "astro check",
    "build": "astro check && astro build",
    "preview": "astro preview"
  }
}
```

每次重要修改至少執行：

```powershell
pnpm check
pnpm build
```

### 5.2 Prettier

用途：統一 Astro、TypeScript、CSS 與 Markdown 格式。

建議加入：

- `prettier`
- `prettier-plugin-astro`

Prettier 只處理格式，不負責判斷程式邏輯正確與否。

### 5.3 ESLint

第一階段暫不加入。先以 TypeScript strict、Astro Check、Prettier 和人工 review 維持品質。

等 TypeScript 程式量明顯增加後，再決定是否加入 ESLint，避免初始化階段出現過多設定與版本整合問題。

## 6. 瀏覽器與測試工具

第一階段不先安裝 Vitest、Playwright 或 Cypress。先使用：

| 工具 | 用途 |
|---|---|
| Chrome DevTools | DOM、CSS、Network、Performance、手機尺寸模擬 |
| Lighthouse | 效能、無障礙、SEO 與 Best Practices 初步檢查 |
| Edge | Chromium 環境交叉確認 |
| Firefox | 檢查 CSS、scroll snap、自訂游標與圖片行為差異 |
| 實體手機 | 確認觸控、字級、動畫效能與 viewport |

等核心互動確定後，第二階段再導入 Playwright 做跨頁流程測試。

## 7. 設計與素材工具

這些不會進入網站執行環境，與 Astro 沒有套件相容性問題：

| 工具 | 必要性 | 用途 |
|---|---|---|
| Figma | 建議 | Sitemap、wireframe、桌機／手機畫面、設計 token |
| Photoshop／Affinity Photo／Photopea | 至少選一個 | 去背、分層 Hero、圖片裁切與壓縮 |
| SVG 編輯器（Illustrator／Inkscape） | 選配 | Logo、裝飾、圖示與向量素材 |
| Squoosh 或同類圖片壓縮工具 | 選配 | 原始素材初步壓縮；最終仍交由 Astro 產生尺寸 |

第一階段不使用 After Effects、Lottie、Rive、Blender 或 Three.js。

## 8. AI 協作工具與規則

採用：

- GitHub Copilot：VS Code 內的即時建議。
- Codex：規劃、跨檔案實作、檢查、測試與問題診斷。

專案需要建立清楚的 AI 指示文件，至少規定：

1. 使用 Astro、HTML、CSS、TypeScript，不使用 React／Vue／Svelte。
2. 只使用 pnpm。
3. 不得自行安裝依賴。
4. 安裝依賴前先說明原生方式為何不足。
5. 不得自行升級 Astro、TypeScript 或 Node major version。
6. 不得刪除 `pnpm-lock.yaml`。
7. 不得用 `any` 或 `@ts-ignore` 隱藏錯誤。
8. 先修改既有元件，不建立功能重複的第二套元件。
9. 修改後必須執行 `pnpm check` 與 `pnpm build`。
10. 保留無 JavaScript或動畫失敗時仍可閱讀的基本內容。

建議文件：

```text
AGENTS.md                  Codex 和其他代理的 repository 規則
.github/copilot-instructions.md   Copilot repository 指示
README.md                  人類維護者的安裝與操作方式
```

## 9. 第一階段明確不安裝的項目

| 工具／服務 | 第一階段決定 | 重新評估條件 |
|---|---|---|
| React | 不安裝 | 出現原生 DOM 狀態難以維護的複雜元件 |
| Vue／Svelte | 不安裝 | 原則上不採用，避免多框架混雜 |
| GSAP | 不安裝 | Hero 分層時間軸確認超出 CSS 可維護範圍 |
| Embla Carousel | 不安裝 | 原生 Scroll Snap 無法滿足 loop、拖曳或進階狀態 |
| Astro ClientRouter | 不啟用 | 所有原生互動穩定後再測試頁面轉場 |
| Tailwind／Bootstrap | 不安裝 | 目前沒有必要 |
| Cloudflare Pages | 不設定 | 本機正式版可成功 build、準備公開時 |
| Cloudflare R2 | 不設定 | 出現大於 25 MiB 或大量可下載媒體時 |
| CMS／資料庫 | 不安裝 | 多人非技術編輯需求確定時 |
| Vitest／Playwright | 不安裝 | 核心互動規格穩定後 |

## 11. 第一階段執行順序

### Step 1：環境準備（半天）

- 安裝 Node.js 24 LTS。
- 確認 `node --version`、`pnpm --version`、`git --version`。
- 安裝 Astro 官方 VS Code extension。

### Step 2：初始化與鎖定規則（半天）

- 執行 `pnpm create astro@latest`，選 minimal 與 strict TypeScript。
- 建立 scripts、格式設定、`.editorconfig`、AI 指示和 README。
- 執行第一次 `pnpm check`、`pnpm build` 並提交 Git。

### Step 3：設計基礎（1～2 天）

- 定義色彩、字體、間距、斷點、圓角與動畫時間。
- 建立全站 Layout、Header、Footer、Container、Button、Card。
- 完成桌機與手機基本排版。

### Step 4：內容系統（1～2 天）

- 建立 Content Collections schema。
- 放入 3 筆真實或 placeholder 活動 Markdown。
- 自動產生活動列表和動態詳情頁。

### Step 5：頁面原型（3～5 天）

- 首頁。
- 活動列表。
- 活動詳情。
- 關於本站。
- 404。

### Step 6：原生互動 prototype（2～4 天）

- 手機選單。
- CSS Scroll Snap Carousel。
- Lightbox。
- Scroll reveal。
- 簡單分層 Hero 入場。
- reduced-motion fallback。

### Step 7：第一階段 QA（1～2 天）

- Astro Check 與 production build。
- Chrome／Edge／Firefox。
- 360、768、1024、1440 px。
- 鍵盤操作、圖片 alt、焦點樣式。
- Lighthouse 初步檢查。

總估計：一位熟悉前端的開發者約 8～16 個工作天；如果同時學習 Astro、TypeScript 與 RWD，建議預留 3～5 週。

## 12. 第一階段最後的技術評估

第一階段完成後，再用實際 prototype 回答四個問題：

1. CSS 動畫是否已足夠？不足才加入 GSAP。
2. 原生 Carousel 是否已足夠？不足才加入 Embla。
3. 原生 TypeScript 是否出現難以維護的複雜狀態？確實出現才加入 React。
4. 網站是否準備公開、是否有大型媒體？到那時再討論 Cloudflare Pages／R2。

這個決策方式能確保每個新工具都是解決已經證明存在的問題，而不是因為預想未來可能需要就先安裝。
