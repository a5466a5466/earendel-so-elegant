# 厄倫蒂兒粉絲應援網站－開發計畫書

版本：規劃草案 v1.0  
日期：2026-07-13  
專案階段：需求與技術規劃（尚未進入製作）

## 1. 執行摘要

本網站完全做得到，而且非常適合採用「靜態內容網站＋局部互動」架構。網站的核心是展示粉絲應援活動成果，包括相片、影片、音訊、活動故事與社群貼文；不需要會員、付款或複雜後台時，約 90% 可以是前端，只有部署、媒體儲存與日後可能加入的內容管理介面需要雲端服務。

建議主方案：

- 網站框架：Astro
- 程式語言：HTML、CSS、TypeScript；必要時局部使用 React 元件
- 內容來源：第一版使用 Astro Content Collections（Markdown／JSON）
- 動畫：CSS 與 Astro View Transitions 為主；複雜時間軸再使用 GSAP
- Carousel：Embla Carousel
- 媒體：圖片由 Astro 最佳化；影片優先放 YouTube；大型原始檔放 Cloudflare R2
- 部署：GitHub 儲存程式碼，Cloudflare Pages 發佈網站，Cloudflare R2 存大型媒體
- 開發工具：Visual Studio Code、Git、Node.js、pnpm、Figma

在素材與視覺方向齊全的前提下，一位有經驗的前端開發者完成「有設計感、響應式、動畫完整、可正式上線」的版本，合理工期約為 5～8 週。只做 MVP 約 3～4 週；若加入大量客製插畫、Rive／WebGL 動畫或內容後台，約 8～12 週。

## 2. 專案目標與邊界

### 2.1 主要目標

1. 集中保存並展示歷次粉絲應援活動成果。
2. 用符合厄倫蒂兒形象的視覺、圖層與動畫營造沉浸感。
3. 讓訪客可快速瀏覽照片、播放影片／音訊，並前往原始社群貼文。
4. 手機、平板、桌機都能正常使用，低階裝置也不會因動畫卡頓。
5. 建立可持續新增活動的內容結構，避免每次更新都重寫頁面。

### 2.2 第一版不建議納入

- 會員註冊、登入、留言板與即時聊天室
- 自建影片串流服務
- 即時同步所有 X／Threads 貼文
- 大型 3D 場景或全站 WebGL
- 複雜資料庫與客製管理後台

上述功能都能做，但會明顯增加後端、安全、維運與審核成本，不符合本案「展示型、90% 前端」的定位。

## 3. 建議網站資訊架構

### 3.1 頁面

| 頁面 | 用途 | 主要內容 |
|---|---|---|
| 首頁 | 建立第一印象與導覽 | 分層 Hero、最新／精選活動、照片 Carousel、精選影片、社群入口 |
| 應援企劃 | 瀏覽所有成果 | 依年份、活動類型或標籤篩選的活動卡片 |
| 活動詳情 | 完整記錄單次活動 | 背景故事、日期、主視覺、照片集、影片、音訊、參與名單與外部連結 |
| 媒體藝廊 | 快速瀏覽媒體 | 圖片瀑布流／網格、Lightbox、影片與音訊分類 |
| 社群牆 | 整理相關貼文 | 精選 X、Threads、YouTube 貼文或連結卡片 |
| 關於本站 | 說明網站性質 | 粉絲非官方聲明、製作團隊、素材來源、版權與聯絡方式 |
| 404 頁 | 錯誤導引 | 角色化文案、返回首頁按鈕、小型動畫 |

第一版可以先做「首頁、企劃列表、活動詳情、關於本站」四種頁型，再於第二階段拆出獨立藝廊與社群牆。

### 3.2 活動資料格式

每個活動應有固定欄位，方便之後自動產生頁面：

```yaml
title: 活動名稱
slug: event-url-name
date: 2026-01-01
summary: 簡短說明
cover: 封面圖片
theme: 主題色或樣式名稱
tags: [生日應援, 粉絲創作]
gallery: [圖片清單]
videos: [YouTube ID 或影片網址]
audio: [音訊清單]
socialPosts: [X／Threads 貼文網址]
credits: [參與者與分工]
featured: true
```

Astro Content Collections 可以對這些欄位做型別檢查，素材漏填或日期格式錯誤時，能在建置階段發現，而不是上線後才壞掉。

## 4. 技術方案

### 4.1 為什麼主推 Astro

Astro 的 Islands 架構會把大部分頁面輸出成靜態 HTML，只有 Carousel、Lightbox、音訊播放器等真正需要互動的區域才載入 JavaScript。因此它特別適合內容展示型網站，並能避免為了幾個動畫把整站做成沉重的單頁應用程式。

優點：

- 預設靜態輸出，速度快、SEO 佳、主機成本低。
- 可使用 `.astro` 元件，也能局部加入 React、Vue 或 Svelte。
- Content Collections 適合管理一系列格式一致的活動資料。
- 內建圖片處理與 View Transitions。
- 未來若需要表單、CMS 或少量 API，仍可擴充成混合／動態網站。

缺點：

- 生態系與人才數量不如 React／Next.js 大。
- 啟用客戶端頁面切換後，部分第三方 script 需要在 `astro:page-load` 重新初始化。
- 若日後變成以登入、即時資料與高度狀態互動為核心的 Web App，Next.js 會更順手。

### 4.2 框架比較

| 方案 | 優點 | 缺點 | 本案適合度 |
|---|---|---|---|
| Astro | 靜態快、JavaScript 少、內容集合與圖片處理方便 | 相較 React，部分套件範例較少 | 最推薦 |
| Next.js＋React | 生態系最大、全端與 CMS 整合成熟、未來擴充強 | 對純展示站較複雜，容易送出過多 JS；靜態匯出有部分限制 | 若確定未來會有會員／後台則考慮 |
| Vite＋React | 開發體驗直接、動畫與 UI 套件多 | 路由、SEO、內容生成與圖片策略需自行組裝 | 可用，但不是最省工 |
| 純 HTML／CSS／JS | 最單純、幾乎零框架成本 | 活動數量增加後難維護，資料與頁面容易重複 | 只適合單頁或短期活動頁 |
| WordPress | 非工程人員容易更新、有成熟後台 | 外掛維護、安全、效能與客製動畫成本較高 | 編輯者很多且必須有後台時才選 |
| Webflow／Framer | 視覺製作快、設計師容易操作 | 長期費用、平台綁定、複雜內容與客製程式受限 | 快速形象頁可用，不適合作為主要內容庫 |

結論：以目前需求使用 Astro，不先加入 React。只有在現成互動元件確實需要 React 時，再把該元件做成 Astro Island。

### 4.3 語言與工具

| 項目 | 建議 | 理由 |
|---|---|---|
| HTML | Astro 模板產生語意化 HTML | SEO、無障礙與靜態輸出 |
| CSS | 原生 CSS＋CSS variables；必要時 SCSS | 高度客製視覺比套版 class 更容易控制 |
| JavaScript | TypeScript strict mode | 降低資料欄位、DOM 與元件介面的錯誤 |
| 套件管理 | pnpm | 安裝快、磁碟使用較有效率 |
| IDE | VS Code | 免費，Astro／TypeScript／ESLint／Prettier 支援完整 |
| 版本控制 | Git＋GitHub | 備份、協作、預覽部署與回復版本 |
| UI／版型 | Figma | 先確認桌機與手機稿，降低邊做邊改的成本 |
| 圖片處理 | Photoshop／Affinity Photo；簡易工作可用 Photopea | 分層素材、去背與 Web 輸出 |
| 動畫素材 | After Effects＋Lottie 或 Rive（選配） | 適合角色化、可重複播放的小型動畫 |

不建議一開始加入 Tailwind。這是高度品牌化而非典型後台介面，原生 CSS 配合設計 token 較清晰；若開發團隊本來就熟悉 Tailwind，也可改用，不影響主架構。

## 5. 動畫與互動設計

### 5.1 動畫分級

| 等級 | 實作方式 | 適用內容 | 成本／風險 |
|---|---|---|---|
| A：基本 | CSS transition、keyframes、Intersection Observer | 淡入、位移、hover、卡片揭露 | 最輕、優先使用 |
| B：編排 | GSAP timeline／ScrollTrigger | 首頁分層入場、視差、依序揭露 | 控制力強，但需維護生命週期 |
| C：向量互動 | Rive 或 Lottie | 角色小動作、圖示、循環裝飾 | 需額外動畫製作流程與 runtime |
| D：3D／特效 | Three.js／WebGL | 立體空間、粒子、著色器 | 體積與效能成本高，第一版不建議 |

建議第一版以 A 為基礎，只把首頁 Hero 與一至兩個重點段落提升到 B。動畫越多不一定越好，應建立「主角動畫、導引動畫、裝飾動畫」的優先順序。

### 5.2 分層入場動畫

首頁主視覺可拆成：背景漸層／紋理、遠景裝飾、中景光效、角色主圖、前景花瓣或粒子、標題與按鈕。每層使用透明 PNG、WebP、AVIF 或 SVG，透過 `transform` 與 `opacity` 編排時間軸：

1. 背景先顯示，避免空白畫面。
2. 遠景與光效緩慢淡入。
3. 角色由小幅位移與縮放進場。
4. 標題和 CTA 依序揭露。
5. 前景只做低幅度循環，不持續大面積移動。

關鍵限制：行動裝置減少圖層、解析度與視差幅度；不能等整段動畫播完才讓使用者操作；重新造訪可縮短或跳過開場。

### 5.3 自訂滑鼠

可用 CSS `cursor: url(...) x y, pointer` 實作。建議：

- 準備 32×32 PNG／SVG，並設定正確 hotspot。
- 保留 `pointer`／`auto` fallback。
- 連結、文字輸入框仍保留可理解的游標語意。
- 觸控裝置完全停用自訂游標。
- 若做「跟隨游標的蒂兒小圖」，它應是獨立裝飾層，使用 `pointer-events: none` 與 `requestAnimationFrame`，且在低效能或 reduced-motion 狀態停用。

Firefox 與 Chromium 通常限制最大 128×128，但實務仍建議 32×32，避免圖片被瀏覽器忽略。

### 5.4 Carousel、Lightbox 與音訊

- Carousel：使用 Embla Carousel；支援滑鼠拖曳、觸控、鍵盤、分頁按鈕與循環控制。
- 不強制自動輪播；若自動輪播，必須提供暫停，並在分頁離開可視區時停止。
- 圖片點擊後使用 Lightbox，保留 alt 文字、焦點鎖定、Esc 關閉與上一張／下一張鍵盤操作。
- 音訊先採 HTML `<audio>`；若要品牌化播放器，再包裝外觀，但不移除原生可用功能。
- 多首歌曲不要同時預載完整檔案，僅載 metadata，點擊後才載音訊。

### 5.5 頁面切換

Astro View Transitions 可提供跨頁淡入、滑動或共享元素效果，也能在不支援的瀏覽器降級。可讓活動卡片的封面自然延伸到詳情頁，但不應每個元素都飛動。

所有動畫必須尊重 `prefers-reduced-motion: reduce`，並在網站設定提供「減少動態效果」切換；減少動態時改用瞬間切換或單純淡入。

## 6. 社群與影音整合

### 6.1 X 與 Threads

兩者都可嵌入公開貼文：X 使用官方 embedded post／oEmbed 與 widget script；Threads 可透過官方 embed script／oEmbed 產生貼文區塊。

但應注意：

- 第三方 script 會增加載入時間，也可能建立 cookie 或追蹤請求。
- 廣告阻擋器、隱私設定、貼文刪除或帳號轉私人後，嵌入可能空白。
- 官方嵌入樣式可客製程度有限，可能與全站風格不完全一致。
- 大量 embed 不應在首屏一次載入。

建議採「靜態預覽卡＋點擊載入官方 embed」：頁面先顯示本站設計的卡片、作者、摘要與縮圖；使用者同意／點擊後才載入官方元件。若使用貼文截圖，必須保留作者、原文連結並確認使用權，不可冒充原生貼文。

### 6.2 YouTube

YouTube 可用 iframe 或 IFrame Player API 嵌入單支影片與播放清單。建議先顯示縮圖 facade，點擊播放後才建立 iframe，可避免首頁一開始載入多個播放器。播放器維持 16:9、至少符合官方最低尺寸，保留控制項、鍵盤與全螢幕功能。

### 6.3 自有影片與音訊

- 已在 YouTube 的影片：直接嵌入，不再複製一份到網站主機。
- 需要保留原始畫質或下載：放 R2，另提供經壓縮的 Web 版本。
- 短背景影片：輸出 WebM＋MP4 fallback，靜音、`playsinline`，並提供 poster；reduced-motion 狀態只顯示 poster。
- 音訊：優先 MP3／AAC，視需求加 Ogg；顯示曲名、作者、長度與授權資訊。

## 7. 媒體、主機與內容管理

### 7.1 建議媒體策略

| 類型 | 儲存位置 | 建議格式／策略 |
|---|---|---|
| Logo、圖示、小型 SVG | Git 專案 | SVG，確認可安全內嵌 |
| 活動圖片 | 專案內或 R2 | AVIF／WebP＋必要 JPG fallback；產生響應式尺寸 |
| 首頁分層素材 | 專案內 | 透明 WebP／PNG；嚴控尺寸與圖層數 |
| 長影片 | YouTube | 使用縮圖 facade 後載入播放器 |
| 可下載原始檔 | R2 | 與網站程式碼分離，設定快取與檔名規則 |
| 音訊 | R2 或專案內少量檔案 | 不預載完整內容 |

Cloudflare Pages 免費方案單一靜態檔案上限為 25 MiB；較大的影片或音訊應放 R2。Pages 適合網站程式和小型資源，R2 適合大量非結構化媒體，兩者用途不要混在一起。

### 7.2 內容管理三階段

1. 第一版：Markdown／JSON＋Git。最穩定、成本最低，由懂基本 Git 的維護者更新。
2. 第二版：加入 Pages CMS、Decap CMS、Keystatic 或類似 Git-based CMS，讓編輯者透過表單新增活動。
3. 第三版：只有多人編輯、草稿審核、排程發佈成為必要需求時，才導入 Sanity／Contentful 等 Headless CMS。

避免第一天就建 CMS；先用三至五筆真實活動資料驗證欄位，否則很容易把錯誤的內容流程永久化。

### 7.3 部署比較

| 平台 | 優點 | 缺點 | 建議 |
|---|---|---|---|
| Cloudflare Pages＋R2 | CDN、預覽部署、靜態站與大型資源可拆分，R2 直接對外傳輸無 egress 費 | 控制台產品較多，首次設定比 GitHub Pages 多 | 最推薦 |
| Vercel | Git 整合與前端預覽體驗很好，Next.js 最順 | 本案若用 Astro，優勢沒有那麼明顯；高流量需注意方案限制 | 次選 |
| Netlify | 靜態站、表單、預覽與整合成熟 | 需注意流量與建置額度 | 次選 |
| GitHub Pages | 免費、簡單 | 大媒體、進階 headers、預覽流程與動態功能彈性較弱 | 原型或極簡版 |

預期初期可使用免費方案，主要固定成本通常是網域名稱；真正的變動成本來自原始圖片／音訊／影片容量與流量，應在素材盤點後估算。

## 8. 視覺與素材前置需求

開工前需取得或製作：

- 官方／授權可用的角色立繪、Logo、代表色與字體規範
- 自訂游標圖案與 favicon
- Hero 分層素材（每層獨立透明檔）
- 至少三個真實活動的標題、日期、說明、封面與照片
- YouTube、X、Threads 原始網址
- 音訊檔、作者、授權方式與是否允許下載
- 粉絲站非官方聲明、素材署名與聯絡方式

必須確認厄倫蒂兒／所屬單位的二創與商標使用規範。參加者照片、聲音、暱稱及社群貼文也要取得適當同意；若照片包含未成年人或可辨識個資，需特別處理。網站應保留素材來源與撤下聯絡機制。

## 9. 效能、無障礙與品質標準

### 9.1 建議驗收指標

- Lighthouse Performance、Accessibility、Best Practices、SEO：主要頁面目標 90 分以上。
- Core Web Vitals（第 75 百分位）：LCP ≤ 2.5 秒、INP ≤ 200 毫秒、CLS ≤ 0.1。
- 首頁初始載入不放原始大圖、不直接載入多個社群 embed／YouTube iframe。
- 圖片都有尺寸、`srcset`、lazy loading（首屏主圖除外）與適當 alt。
- 鍵盤可以操作選單、Carousel、Lightbox、播放器和所有連結。
- 顏色對比、焦點樣式、標題階層與語意標籤符合 WCAG 2.2 AA 的實務要求。
- 支援目前主流 Chrome、Edge、Firefox、Safari，以及 iOS Safari／Android Chrome。

### 9.2 測試

- 單元測試：內容 schema、資料轉換、關鍵工具函式。
- 元件測試：Carousel、篩選、Lightbox、音訊播放器。
- E2E：首頁到活動詳情、媒體開啟、外部社群連結、404。
- 視覺測試：至少 360、768、1024、1440 px 寬度；Windows／macOS 字型差異。
- 效能測試：一般手機節流、首次造訪、快取後造訪、reduced-motion。
- 失敗情境：X／Threads 被阻擋、YouTube 不可播放、圖片 404 時仍有可理解的 fallback。

## 10. 開發階段與時程

以下以一位有經驗的前端開發者、素材大致齊全、回饋可在 1～2 天內完成為前提。

| 階段 | 工作內容 | 估計工作天 |
|---|---|---:|
| 0. 素材與需求盤點 | 內容清單、授權、參考網站、成功標準 | 2～3 |
| 1. 資訊架構與線框 | Sitemap、頁型、內容 schema、mobile-first wireframe | 2～4 |
| 2. 視覺設計 | Moodboard、設計 token、首頁與詳情頁桌機／手機稿 | 5～8 |
| 3. 技術基礎 | Astro、TypeScript、lint、內容集合、部署預覽 | 2～3 |
| 4. 核心頁面 | Layout、導覽、首頁、列表、詳情、關於與 404 | 5～8 |
| 5. 媒體互動 | Carousel、Lightbox、YouTube facade、音訊、社群卡 | 4～6 |
| 6. 動畫 | Hero 時間軸、scroll reveal、頁面轉場、自訂游標 | 4～8 |
| 7. 內容上稿 | 圖片最佳化、活動資料、署名與連結校正 | 3～6 |
| 8. QA 與上線 | 響應式、無障礙、效能、SEO、分析、正式網域 | 3～5 |
| **合計** | 部分工作可交錯進行 | **30～51 工作天** |

實際曆時建議：

- MVP：3～4 週。4 個頁型、3 個活動、基本動畫、YouTube 與照片展示。
- 建議正式版：6～8 週。完整視覺、分層 Hero、媒體體驗、社群 fallback、效能與無障礙 QA。
- 進階版：8～12 週。大量客製動畫、Rive、CMS、多語系或大量內容移轉。
- 若由前端初學者邊學邊做：預留 2～4 個月較合理。

最大變數通常不是寫程式，而是主視覺定稿、分層素材製作、內容整理、授權確認與回饋輪次。

## 11. 里程碑與驗收

### M1：方向確認

- 確認網站定位、頁面清單、主色、風格關鍵字與參考網站。
- 確認至少三個真實活動資料與素材可用。

### M2：可點擊原型

- 桌機與手機首頁、列表、詳情頁可操作。
- 動畫以簡化 prototype 展示節奏，尚不追求最終畫質。

### M3：功能完成

- 所有頁面、Carousel、Lightbox、影片、音訊、社群卡與內容 schema 可用。
- 預覽網址可供手機實機審查。

### M4：視覺與內容完成

- 正式素材與文案上稿，動畫、fallback、reduced-motion 完成。
- 完成素材署名、非官方聲明與隱私／第三方內容說明。

### M5：正式上線

- 通過響應式、鍵盤、跨瀏覽器、Lighthouse 與連結檢查。
- 綁定網域、建立備份／回復方式與更新操作說明。

## 12. 主要風險與對策

| 風險 | 影響 | 對策 |
|---|---|---|
| 首頁圖層過多／檔案過大 | 手機載入慢、動畫掉幀 | 行動版減層、輸出 AVIF／WebP、首屏資源預算與實機測試 |
| 社群貼文失效或被封鎖 | 留白、版面跳動 | 靜態預覽卡、保留原文連結、固定預留高度 |
| 動畫太強 | 不適、可讀性降低 | reduced-motion、站內開關、只在重點段落使用 |
| 內容格式不一致 | 每個活動頁都要特例 | 先定 schema，用真實資料驗證後再大量上稿 |
| 素材權利不明 | 下架或爭議 | 上稿前記錄作者、來源、授權與撤下方式 |
| 大檔案進 Git | 儲存庫膨脹、部署失敗 | 大型影片／音訊／原圖放 R2，不提交 Git |
| 過早加入 CMS／3D | 時程膨脹 | 先完成靜態 MVP，以數據與維護需求決定第二階段 |

## 13. 建議第一版範圍

為了在合理時間內交付一個好看的版本，第一版建議鎖定：

- 首頁、企劃列表、企劃詳情、關於、404。
- 3～6 個完整應援活動。
- 分層 Hero 入場、基本 scroll reveal、卡片共享元素轉場。
- 自訂 CSS 游標；桌機才啟用，保留 fallback。
- 1 個 Carousel、1 套 Lightbox、YouTube facade、HTML 音訊播放器。
- X／Threads 使用靜態預覽卡，點擊或同意後載入官方嵌入。
- Astro Content Collections，不做 CMS。
- Cloudflare Pages 上線；大型媒體才啟用 R2。

這個範圍已能完整表現「粉絲應援成果館」，同時把最昂貴的 3D、即時資料與後台留待確定需要時再做。

## 14. 下一步決策清單

進入設計／開發前，需要依序確認：

1. 網站是否為長期官方／非官方粉絲站，以及使用的正式名稱與網域。
2. 視覺關鍵字與 3～5 個參考網站：夢幻、科技、舞台、可愛、神秘等方向的比重。
3. 可合法使用的角色圖、Logo、字體、照片、影片與音訊。
4. 第一版要展示的 3～6 個活動及每個活動負責整理資料的人。
5. 維護者是否會 Git；若不會，第二階段是否需要 Git-based CMS。
6. 動畫預算選擇：基本、精緻或進階互動。
7. 是否需要繁中以外的日文／英文版本。

完成上述清單後，下一份產出應是「網站需求規格 v1＋Sitemap＋首頁／活動詳情線框稿」，確認後才開始建立 Astro 專案。
