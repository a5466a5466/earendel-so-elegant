## 1. 企劃摘要

本企劃將在正式網站之外建立獨立的 `/lab/` 功能實驗區，逐項測試 Astro、原生 HTML／CSS／TypeScript、瀏覽器 API 與必要第三方服務能提供的效果。

實驗的目的不是把所有效果都塞進正式網站，而是建立一套可重複使用的「功能能力表」與「實際工時基準」，以便未來規劃應援活動時，能依照素材、裝置、品質和時程做出可靠估算。

實驗完成後，每項功能應得到以下其中一種結論：

- **採用**：體驗、效能與維護成本均適合正式網站。
- **條件採用**：只在桌機、活動特頁或特定素材齊全時採用。
- **保留研究**：技術可行，但目前成本或風險過高。
- **不採用**：效果不符合網站定位，或成本明顯高於價值。

## 2. 核心目標

1. 驗證不同互動與媒體功能在 Astro 的實作方式。
2. 比較原生方案與第三方套件的差異，不先假設一定需要套件。
3. 找出最適合厄倫蒂兒形象的視覺風格與互動節奏。
4. 在桌機、平板與手機上實際測試 RWD、觸控和效能。
5. 記錄 Prototype 與正式完成所需的實際工時。
6. 分離「程式開發工時」與「素材製作工時」，避免低估專案。
7. 建立未來活動網站可重用的元件、測試規格和估時依據。

## 3. 不在本企劃直接完成的事項

- 不因研究而立即改動正式首頁。
- 不一次安裝所有可能用到的套件。
- 不把 `/lab/` 當成正式網站內容對外發布。
- 不在尚未驗證前承諾 Live2D、WebGL 或即時社群同步一定納入正式版。
- 不在本階段設定 Cloudflare Pages、R2、CMS 或正式網域。
- 不使用權利不明的角色、音樂、照片、影片或社群內容作為正式素材。

## 8. 各功能實驗規格

### F01　動畫滑鼠

- **能做到的功能**：替換一般／連結／按下狀態游標；加入光點、星屑、拖尾或角色跟隨；可依區塊改變外觀。
- **使用者體驗**：適合增加品牌感，但不能遮擋點擊目標、文字選取或輸入游標。主效果只應出現在精準滑鼠裝置。
- **預計時間**：靜態自訂游標 0.5 天；跟隨與拖尾 Prototype 1～2 天；正式整合 2～4 天。
- **準備素材**：32×32 或 48×48 PNG／SVG 游標、hotspot 定義、跟隨角色或粒子素材。
- **RWD**：手機和平板觸控停用；`pointer: coarse` 時回到原生介面。
- **額外套件**：第一版不需要；CSS `cursor`、Pointer Events、`requestAnimationFrame` 足夠。
- **可行性**：高。
- **主要風險**：瀏覽器游標尺寸限制、拖尾造成掉幀、降低可用性。
- **驗收**：不影響連結、選字、輸入框；觸控裝置無殘留；reduced-motion 可停用。

### F02　網站內桌寵

- **能做到的功能**：角色停留在頁面角落、走動、待機、點擊反應、拖曳、對話泡泡、記住位置與開關狀態。
- **能力邊界**：一般網站只能在瀏覽器頁面內活動，不能在使用者 Windows 桌面上跨應用程式行走。真正的作業系統桌寵需要 Electron、Tauri 或原生桌面程式，屬另一個專案。
- **使用者體驗**：可愛且有記憶點，但必須提供關閉／暫停，不得遮住導覽、文字或手機畫面。
- **預計時間**：基本 DOM 桌寵 1～3 天；具狀態、拖曳、持久化與完整 QA 為 5～10 天；真桌寵約 2～6 週起。
- **準備素材**：待機、走路、點擊、睡眠等狀態素材；透明序列、sprite sheet、Rive、Lottie 或 Live2D 模型。
- **RWD**：桌機完整；平板簡化；手機預設收合成按鈕或停用。
- **額外套件**：原生 DOM Prototype 不需要；複雜狀態可再評估 XState、Rive 或 UI Island。
- **可行性**：網站內中高；真正桌面寵物低於本專案範圍。
- **主要風險**：素材量、狀態爆增、遮擋、持續動畫耗電。
- **驗收**：可關閉、可暫停、不遮擋核心內容、重新進頁能正確恢復或重設。

### F03　Navigation

- **能做到的功能**：單頁錨點、多頁路由、Sticky Header、手機選單、目前區塊提示、麵包屑與返回上一頁。
- **使用者體驗**：使用者始終知道自己在哪裡、下一步去哪裡；返回鍵與深層連結必須符合瀏覽器預期。
- **預計時間**：Prototype 0.5～1 天；正式版 2～4 天。
- **準備素材**：頁面架構、導覽名稱、Logo、圖示與手機選單稿。
- **RWD**：桌機橫向、手機抽屜或全畫面選單；避免只靠 hover。
- **額外套件**：不需要，使用 Astro file-based routing 與原生 TypeScript。
- **可行性**：高。
- **驗收**：鍵盤可完整操作；焦點順序正確；直接開啟 `#section` 或子頁網址有效。

### F04　Page Transition

- **能做到的功能**：MPA 原生 View Transition、淡入／滑動、卡片封面共享到詳情頁、返回方向動畫；必要時測試 Astro `<ClientRouter />`。
- **使用者體驗**：提升跨頁連續感，但不能讓切換變慢或使返回鍵行為異常。
- **預計時間**：基本 Prototype 0.5～1 天；共享元素與 ClientRouter 比較 1～2 天；正式版 2～5 天。
- **準備素材**：至少兩個獨立頁面、相同封面或角色元素、轉場分鏡。
- **RWD**：行動版縮短時間、減少位移；reduced-motion 使用瞬間切換或短淡入。
- **額外套件**：Astro 內建能力即可，不需新增 dependency。
- **可行性**：高。
- **主要風險**：ClientRouter 會改變 script 與第三方 embed 的生命週期，可能需要在 `astro:page-load` 重新初始化。
- **驗收**：前進、返回、重新整理、直接進入子頁皆正確；未支援瀏覽器能正常換頁。

### F05　Scroll Animation

- **能做到的功能**：淡入、位移、逐字揭露、Sticky Story、視差、進度條、橫向段落。
- **使用者體驗**：用於建立敘事與視線引導；不應讓閱讀必須等待動畫完成。
- **預計時間**：基本 reveal 1 天；多段 Prototype 1～2 天；正式版 3～7 天；複雜敘事另計。
- **準備素材**：分層圖片、段落順序、動畫分鏡與手機簡化稿。
- **RWD**：手機減少視差與 sticky 長度；橫向滾動不能阻斷垂直手勢。
- **額外套件**：先用 CSS、Intersection Observer、Scroll-driven Animations；原生不足才評估 GSAP ScrollTrigger。
- **可行性**：高。
- **主要風險**：暈動症、低階手機掉幀、內容高度改變造成觸發錯誤。
- **驗收**：reduced-motion 完整可讀；60 FPS 為目標；不造成明顯 CLS。

### F06　Gallery

- **能做到的功能**：固定網格、Masonry 視覺、分類、年份切換、圖片說明、Lazy Loading。
- **使用者體驗**：大量活動照片能快速掃視並保持一致裁切；重要圖片不應被過度裁切。
- **預計時間**：Prototype 1～2 天；正式整合 3～6 天。
- **準備素材**：至少 12～30 張不同長寬比圖片、縮圖、alt、作者和授權資訊。
- **RWD**：桌機 3～4 欄、平板 2～3 欄、手機 1～2 欄；觸控目標足夠大。
- **額外套件**：CSS Grid 先行；只有真正 Masonry 排序需求才評估套件。
- **可行性**：高。
- **驗收**：圖片載入不造成版面跳動；失敗圖片有 fallback；鍵盤能到達可操作圖片。

### F07　Carousel

- **能做到的功能**：拖曳、觸控滑動、上一張／下一張、分頁、循環、置中卡片與縮圖導航。
- **使用者體驗**：適合少量精選內容，不適合隱藏大量重要資訊；不預設強制自動播放。
- **預計時間**：原生 Scroll Snap Prototype 1～2 天；正式版 3～6 天。
- **準備素材**：6～10 張測試圖、每張標題與說明、不同長寬比測試。
- **RWD**：手機顯示 1 張加下一張提示；支援手指滑動；按鈕不得超出畫面。
- **額外套件**：先原生；需要可靠 loop、複雜拖曳和狀態同步時才評估 Embla Carousel。
- **可行性**：高。
- **驗收**：鍵盤、滑鼠與觸控均可用；若自動播放必須能暫停。

### F08　Lightbox

- **能做到的功能**：放大圖片、上一張／下一張、圖片說明、縮圖、下載連結與全螢幕。
- **使用者體驗**：保留瀏覽位置，讓使用者專注看大圖；關閉後焦點應回到原本圖片。
- **預計時間**：Prototype 1～2 天；正式版 3～5 天。
- **準備素材**：縮圖與大圖、alt、caption、作者與下載權限。
- **RWD**：手機支援縮放或滑動；關閉按鈕避開瀏海與系統區域。
- **額外套件**：先用 `<dialog>` 與原生 TypeScript；手勢與縮放需求明確後再評估 PhotoSwipe 等方案。
- **可行性**：高。
- **驗收**：Esc 關閉、焦點鎖定、背景不可誤操作、螢幕閱讀器能理解狀態。

### F09　自有 Video

- **能做到的功能**：短背景影片、活動花絮、poster、字幕、靜音自動播放與自訂外觀。
- **使用者體驗**：背景影片必須安靜、不干擾文字；正式影片保留原生控制與字幕。
- **預計時間**：基本 Prototype 0.5～1 天；正式版 2～4 天，不含轉檔。
- **準備素材**：MP4／WebM、poster、字幕 VTT、授權、不同解析度版本。
- **RWD**：手機可換成 poster 或短低畫質版本；禁止大影片搶首屏頻寬。
- **額外套件**：HTML `<video>` 足夠；複雜串流才需播放器套件。
- **可行性**：高。
- **驗收**：無聲自動播放符合瀏覽器限制；有聲播放必須由使用者觸發；reduced-motion 顯示靜態圖。

### F10　Audio Player

- **能做到的功能**：單曲／播放清單、進度、音量、曲目切換、歌詞或波形、跨頁持續播放實驗。
- **使用者體驗**：所有聲音預設關閉，由使用者啟動；切換曲目時資訊清楚，不同音訊不會同時播放。
- **預計時間**：原生播放器 1 天；品牌化 Prototype 1～2 天；正式版 3～7 天。
- **準備素材**：MP3／AAC、曲名、作者、封面、時長、授權；選配逐字歌詞或波形資料。
- **RWD**：手機控制簡化但仍保留播放／暫停、進度與音量策略。
- **額外套件**：先使用 `<audio>`；只有波形需求才評估 WaveSurfer 等套件。
- **可行性**：高。
- **主要風險**：音檔容量、瀏覽器自動播放限制、音樂授權。
- **驗收**：鍵盤可控制；離開可視區或切頁行為符合規格；一次只播放一個音源。

### F11　Interactive Islands

- **能做到的功能**：在 Astro 頁面局部加入帶狀態的 React／Vue／Svelte 元件，例如角色互動、複雜篩選或小遊戲控制器。
- **使用者體驗**：只有需要互動的區域下載 JavaScript，其餘內容維持靜態快速。
- **預計時間**：比較 Prototype 1～3 天；正式導入單一框架與元件 4～8 天。
- **準備素材**：明確的複雜互動案例；不能只做計數器後就認定需要框架。
- **RWD**：由 Island 自行處理；需測 hydration 前後的版面一致性。
- **額外套件**：會新增對應 Astro integration 與框架 runtime，必須先取得同意。
- **可行性**：高，但目前必要性低。
- **驗收**：比較原生 TypeScript 與 Island 的程式量、JS 大小、維護性後才下結論。

### F12　X 貼文嵌入

- **能做到的功能**：官方 Embedded Post、oEmbed、深色主題、隱藏對話或媒體、靜態預覽卡與點擊載入。
- **使用者體驗**：可直接查看原文與媒體，但樣式受 X 控制，載入可能較慢或被阻擋。
- **預計時間**：單篇直接嵌入 0.5～1 天；含同意、延遲載入、fallback 與多篇管理為 2～4 天。
- **準備素材**：公開貼文 URL、作者、備援文字、原文連結與展示同意／規範確認。
- **RWD**：官方元件可縮放，但高度不可預知；手機需預留穩定容器。
- **額外套件**：不需要；使用 X 官方 `widgets.js`／oEmbed。
- **可行性**：中高。
- **主要風險**：第三方追蹤與同意、廣告阻擋、刪文、轉私人、ClientRouter 下重新初始化。
- **驗收**：被阻擋或刪除時仍顯示可理解的 fallback 與原文連結。

### F13　Threads 貼文嵌入

- **能做到的功能**：公開貼文 embed code、`embed.js`、oEmbed 與靜態預覽卡。
- **使用者體驗**：與 X 類似，可看原貼文但視覺客製有限。
- **預計時間**：單篇 0.5～1 天；批次、延遲載入與 fallback 為 2～5 天。
- **準備素材**：公開 Threads URL、作者、替代摘要與原文連結。
- **RWD**：最小寬度與長文高度需實測；手機避免容器溢出。
- **額外套件**：原則上使用官方 embed／oEmbed，不採用未確認維護的第三方聚合器。
- **可行性**：中；API 權限、端點和平台政策可能變動。
- **主要風險**：第三方 script、帳號／貼文狀態、樣式變動、隱私同意。
- **驗收**：至少測試文字、圖片、影片、刪除與載入失敗情境。

### F14　YouTube 影片嵌入

- **能做到的功能**：單支影片、播放清單、開始時間、字幕、iframe、IFrame Player API 與點擊後載入 facade。
- **使用者體驗**：點擊封面後才載入播放器，能明顯改善首頁初始速度並減少不必要第三方連線。
- **預計時間**：直接 iframe 0.5 天；facade Prototype 0.5～1 天；正式版 2～3 天。
- **準備素材**：影片 ID／URL、合法縮圖、標題、頻道、poster fallback。
- **RWD**：維持 16:9，播放器視窗不得低於官方最低需求；手機保留全螢幕。
- **額外套件**：不需要；直接使用官方 iframe 或 IFrame Player API。
- **可行性**：高。
- **主要風險**：影片禁止嵌入、刪除、年齡限制、Cookie 與第三方請求。
- **驗收**：鍵盤控制、全螢幕、錯誤 fallback、延遲載入與多影片只載入被選中的一支。

### F15　小遊戲

- **能做到的功能**：點擊收集星星、節奏點擊、記憶配對、角色換裝、抽卡式互動、活動彩蛋。
- **使用者體驗**：適合活動限定頁與分享，不應成為瀏覽成果的阻礙。
- **預計時間**：簡單遊戲 Prototype 2～5 天；完整遊戲 1～4 週，視美術、音效、關卡與計分而定。
- **準備素材**：角色、背景、UI、動畫、音效、規則、教學、成功／失敗狀態。
- **RWD**：需同時設計鍵鼠與觸控；橫向遊戲要處理手機方向與安全區域。
- **額外套件**：簡單遊戲用 DOM／Canvas；複雜碰撞與場景才評估 Phaser 等引擎或互動 Island。
- **可行性**：中高。
- **主要風險**：需求容易膨脹；素材與 QA 通常比程式更花時間。
- **驗收**：規則可理解、可重新開始、無聲也能玩、手機至少維持可接受幀率。

### F16　人物動畫／Live2D 類效果

- **能做到的功能**：依成本分四級：
  1. GIF／Animated WebP／短透明影片：固定循環動作。
  2. Sprite Sheet：可切換待機、說話、點擊等狀態。
  3. Lottie／Rive：向量骨架、狀態機、指標或點擊互動。
  4. Live2D Cubism SDK for Web：WebGL 即時模型、呼吸、視線、嘴型與參數混合。
- **使用者體驗**：GIF 最容易但不能自然互動；Rive 適合圖示化角色；真正 Live2D 表現最好但成本最高。
- **預計時間**：GIF／影片整合 1 天；Sprite 2～5 天；Rive／Lottie 整合 2～5 天；Live2D Web Prototype 3～7 天，正式整合 1～3 週。
- **素材時間**：GIF 1～3 天；Rive／Lottie 3～10 天；新的 Live2D 模型拆圖、建模與動作通常 2～8 週以上。
- **準備素材**：分層 PSD 或正式模型、動作、表情、授權、低階裝置替代圖。
- **RWD**：手機減少模型解析度、更新頻率與互動參數；低效能或 reduced-motion 顯示靜態立繪。
- **額外套件**：依方案可能需要 Rive runtime、lottie-web 或 Live2D Cubism SDK；必須另審授權與檔案大小。
- **可行性**：技術中高；素材與授權中低。
- **主要風險**：Live2D 不只是「放一張圖」；需模型素材、WebGL、效能測試與 SDK 發布授權確認。
- **驗收**：先以同一角色比較 GIF、Sprite、Rive 與 Live2D 的畫質、互動、工時與載入成本，再選正式方案。

### F17　按鍵音效

- **能做到的功能**：按鈕 hover／click、頁面完成、遊戲回饋與自訂音量；可建立全站靜音偏好。
- **使用者體驗**：應是可選的細節回饋，不能預設突然出聲或每個操作都播放。
- **預計時間**：Prototype 0.5～1 天；正式版 1～3 天。
- **準備素材**：短音效、音量正規化、授權、至少成功／取消兩類；優先使用短 WAV／OGG／MP3。
- **RWD**：技術上相同；手機需考慮靜音模式與延遲。
- **額外套件**：原生 Web Audio／`<audio>` 足夠；複雜混音才評估 Howler.js。
- **可行性**：高。
- **驗收**：初始靜音或首次互動後才啟用；提供永久關閉；快速連點不造成音量堆疊。

### F18　Content Collections／動態活動頁

- **能做到的功能**：Markdown 活動資料、schema 驗證、自動產生列表／詳情頁、標籤、日期排序與相關活動。
- **使用者體驗**：所有活動頁結構一致，連結可分享，內容不必全部塞在首頁。
- **預計時間**：3 筆假資料 Prototype 1～2 天；正式 schema 與頁型 3～6 天。
- **準備素材**：至少 3 個真實或合法 placeholder 活動的完整欄位。
- **RWD**：主要由活動頁版型負責，資料層本身無裝置限制。
- **額外套件**：Astro 內建，不需要新增 dependency。
- **可行性**：高，建議列為核心基礎。
- **驗收**：缺欄位、錯日期或無效媒體在 build 階段能被發現。

### F19　搜尋、標籤與篩選

- **能做到的功能**：年份、活動類型、媒體類型篩選；關鍵字搜尋；URL 保留篩選狀態。
- **使用者體驗**：內容超過約 10～20 個活動後價值明顯；資料少時避免過度設計。
- **預計時間**：原生篩選 1～2 天；正式搜尋、URL 與無結果狀態 3～7 天。
- **準備素材**：一致的 tags、年份、摘要與搜尋測試詞。
- **RWD**：手機採可收合 filter，並保持觸控目標與結果數提示。
- **額外套件**：先原生；大量內容才評估 Pagefind 等靜態搜尋方案。
- **可行性**：高。
- **驗收**：鍵盤可操作；返回鍵能恢復條件；無 JavaScript 時仍能看完整列表。

### F20　圖片最佳化與 Lazy Loading

- **能做到的功能**：Astro Image、WebP／AVIF、`srcset`、尺寸預留、首屏優先與其餘 lazy loading。
- **使用者體驗**：圖片載入更快、行動網路耗量較低、版面不跳動。
- **預計時間**：比較 Prototype 1～2 天；正式媒體管線 3～6 天，不含人工修圖。
- **準備素材**：同一批高解析原圖、不同裁切焦點、透明與非透明格式。
- **RWD**：這項實驗核心就是為不同 viewport 輸出合理尺寸。
- **額外套件**：優先使用 Astro 內建圖片能力。
- **可行性**：高，正式版必做。
- **驗收**：比較原圖與最佳化後的大小、LCP、CLS、畫質和手機流量。

### F21　粒子、視差與 WebGL 背景

- **能做到的功能**：CSS 星塵、Canvas 粒子、滑鼠斥力、景深、Three.js 3D 空間或 shader。
- **使用者體驗**：適合 Hero 或活動特頁，不能覆蓋全站造成疲勞與耗電。
- **預計時間**：CSS／Canvas Prototype 1～4 天；正式 Canvas 5～10 天；Three.js 1～3 週起。
- **準備素材**：粒子圖、分層背景、色彩與運動規則；3D 則需模型和貼圖。
- **RWD**：手機減少粒子數、解析度和更新頻率，或完全換成靜態背景。
- **額外套件**：CSS／Canvas 不需要；3D 才評估 Three.js。
- **可行性**：中。
- **驗收**：長時間執行不持續增加記憶體；低階手機可自動降級；文字對比保持穩定。

### F22　分享、複製連結與 QR Code

- **能做到的功能**：Web Share API、複製網址、活動深層連結、QR Code、分享成功提示。
- **使用者體驗**：手機優先使用系統分享；桌機提供複製與 QR，方便活動現場使用。
- **預計時間**：Prototype 0.5～1 天；正式版 1～3 天。
- **準備素材**：分享標題、摘要、Open Graph 圖與 QR 樣式。
- **RWD**：手機、桌機採不同 fallback。
- **額外套件**：分享與剪貼簿原生；QR 可用小型套件或 build-time 產生，需另確認。
- **可行性**：高。
- **驗收**：不支援 Web Share 時自動顯示複製連結；成功與失敗都有提示。

### F23　多語系

- **能做到的功能**：繁中／日文／英文路由、語言切換、hreflang、每語言獨立內容。
- **使用者體驗**：擴大觸及，但不應用機器翻譯直接取代正式文案。
- **預計時間**：路由 Prototype 1～3 天；正式架構 5～12 天；翻譯與校對另計。
- **準備素材**：翻譯、語言名稱、日期格式、不同字型與文字長度測試。
- **RWD**：日文／英文換行與按鈕長度需個別測試。
- **額外套件**：先使用 Astro 內建 i18n routing；必要時才加翻譯工具。
- **可行性**：高，但內容成本高。
- **驗收**：每頁語言一致、切換能到相對應頁面、缺翻譯有明確 fallback。

### F24　投稿表單／留言

- **能做到的功能**：投稿、聯絡、祝福留言、圖片上傳、審核後展示。
- **能力邊界**：純靜態 Astro 可顯示表單，但真正接收、儲存、防垃圾與審核需要外部服務或後端。
- **使用者體驗**：適合活動參與；必須顯示用途、授權、個資與成功／失敗狀態。
- **預計時間**：前端 Prototype 1～3 天；正式含服務、驗證、防濫用與審核 1～3 週。
- **準備素材**：欄位、同意文字、隱私政策、錯誤訊息與管理流程。
- **RWD**：mobile-first，輸入類型與鍵盤必須正確。
- **額外套件／服務**：後續評估 Cloudflare Functions、表單服務或資料庫；本階段不安裝。
- **可行性**：中。
- **驗收**：垃圾訊息、重複提交、斷線、檔案限制與刪除請求皆有處理方案。

### F25　PWA／離線瀏覽

- **能做到的功能**：安裝到主畫面、離線頁、快取活動內容、更新提示。
- **使用者體驗**：對活動現場網路不穩可能有價值；普通形象網站則不一定值得增加複雜度。
- **預計時間**：離線頁 Prototype 1～3 天；正式快取策略 4～10 天。
- **準備素材**：多尺寸 App Icon、manifest 名稱、離線內容和更新文案。
- **RWD**：主要是手機安裝體驗，但桌機也需測試。
- **額外套件**：可評估 Vite PWA integration；新增前需確認必要性。
- **可行性**：中高。
- **主要風險**：錯誤快取導致舊內容、媒體容量過大、更新難以理解。
- **驗收**：離線狀態、更新版本、清除快取與失敗恢復都有測試。

### F26　動態偏好與效能模式

- **能做到的功能**：減少動態、關閉音效、低畫質、停止桌寵、停用粒子，並記住使用者選擇。
- **使用者體驗**：讓沉浸效果不犧牲舒適、電量與可及性，是所有進階效果的共用安全層。
- **預計時間**：控制器 Prototype 1～2 天；正式整合 3～6 天。
- **準備素材**：設定介面圖示、說明文字、每項功能的降級版本。
- **RWD**：手機更容易觸發低效能預設；設定面板需適合觸控。
- **額外套件**：不需要，使用 `matchMedia`、`localStorage` 與 CSS data attributes。
- **可行性**：高，建議第一階段建立。
- **驗收**：系統 `prefers-reduced-motion` 與站內選擇不互相衝突；設定可重設。

## 10. F01～F26 詳細實作順序

### 10.1 排序原則

實作順序不依照功能編號排列，而依照下列相依關係排列：

1. **先資料、後畫面**：Content Collections、圖片策略與全站偏好先建立，避免後續元件各自發展一套資料或設定。
2. **先路由、後第三方 script**：Navigation、i18n 與 Page Transition 會影響所有 script 的初始化方式，必須先決定。
3. **先原生、後套件**：Carousel、Lightbox、音訊等先做原生基線，只有實際驗收不足才加入套件。
4. **先共用媒體、後外部平台**：先完成自有圖片／影音容器，再讓 X、Threads、YouTube 共用延遲載入與 fallback。
5. **先可關閉機制、後持續動畫**：動態偏好與效能模式必須早於動畫滑鼠、桌寵、粒子和 Live2D。
6. **先決定動畫 runtime、後做桌寵**：人物動畫格式會直接決定桌寵的狀態與渲染架構。
7. **後端與 Service Worker 最後**：表單會改變部署架構；PWA 快取會干擾所有前期除錯，因此放在最後。

| 批次 | 順序 | 內容 | Prototype 累計 | 必須得到的決策 |

| B：原生內容體驗 | 07～16 | Scroll、Gallery、Carousel、Lightbox、影音、分享、搜尋 | 約 8～16 天 | 哪些原生方案足夠、哪些需要套件 |
| C：外部與裝飾 | 17～19 | X、Threads、動畫滑鼠 | 約 2～5 天 | 第三方內容策略、游標是否適合正式站 |
| D：進階互動 | 20～24 | Islands、人物、桌寵、WebGL、小遊戲 | 約 6～22 天以上 | UI 框架、動畫 runtime、進階功能上限 |
| E：架構擴張 | 25～26 | 表單後端、PWA | 約 2～6 天（僅 Prototype） | 是否值得引入後端與 Service Worker |

若 26 項全部完成 Prototype，初估約 **23～61 個工作天**；人物素材、正式小遊戲、後端與跨瀏覽器正式 QA 不包含在此總數內。每完成一個批次都應安排一次停止檢討，不預設下一批一定執行。

### 10.5 套件決策閘門

為降低依賴衝突，以下候選不能在一開始一起安裝：

| 原生實驗 | 只有在以下不足時才評估 | 候選；一次只選一個 |
|---|---|---|
| F05 CSS／Intersection Observer | 需要複雜時間軸、pinning、精準 scrub | GSAP＋ScrollTrigger |
| F07 CSS Scroll Snap | loop、拖曳狀態與對齊無法可靠完成 | Embla Carousel |
| F08 `<dialog>` | 需要成熟縮放、手勢與大量圖片管理 | PhotoSwipe 類方案 |
| F10 `<audio>`／Web Audio | 需要波形或複雜播放佇列 | WaveSurfer 或 Howler，依需求二選一 |
| F11 原生 TypeScript | 狀態與 DOM 同步已證明難以維護 | React／Vue／Svelte 只選一個 |
| F16 CSS／Sprite | 需要向量狀態機或模型參數 | Rive／Lottie／Live2D 依目標擇一 |
| F21 CSS／Canvas | 需要真正 3D、shader、模型 | Three.js |
| F24 靜態表單 | 確定需要接收、審核與儲存 | 後端服務另案評估 |
| F25 無 Service Worker | 確定需要安裝與離線 | 單一 PWA integration |

### 10.6 每次新增依賴的安全流程

任何 dependency 必須逐一進行，不可一次安裝多個候選：

1. 記錄原生方案具體不足與新套件要解決的問題。
2. 核對套件 License、最近更新、Astro／Node 相容版本、bundle 大小與維護者狀態。
3. 說明替代方案、退出方式與對既有功能的生命週期影響。
4. 取得使用者同意後才使用 `pnpm add`。
5. 一次只新增一個主要套件，保留 `pnpm-lock.yaml` 差異。
6. 立即執行 build，並回歸 Navigation、Page Transition、reduced-motion 與手機版。
7. 若效果不達標，移除該套件並記錄原因，不留下未使用 dependency。
8. 套件通過後才進入下一個功能，避免無法判斷是哪個依賴造成衝突。

### 10.7 特別衝突規則

- 若採用 `<ClientRouter />`，所有第三方 script 必須透過統一 loader 或 `astro:page-load` 初始化，不得在各頁任意重複插入。
- Rive、Lottie 與 Live2D 初次實驗不得同時安裝；先用同一組驗收案例選出一個候選。
- React、Vue、Svelte 不得在同一專案並存測試；F11 只選一個最有理由的框架與原生方案比較。
- GSAP 與瀏覽器原生 Scroll-driven Animations 不應同時控制同一元素。
- Gallery、Carousel、Lightbox 必須共用媒體資料和圖片元件，不各自維護圖片清單。
- YouTube、X、Threads 必須共用第三方同意、延遲載入與 fallback 外殼。
- Audio Player 與按鍵音效必須共用靜音偏好和音訊管理器。
- 桌寵、小遊戲和人物動畫共用同一角色素材策略，不為每個功能輸出不相容格式。
- PWA 啟用前必須完成所有第三方 embed 與媒體快取策略，開發模式預設不註冊 Service Worker。

## 11. 每項實驗交付物

每個功能結束時必須交付：

1. 可直接開啟的 `/lab/...` 示範頁。
2. 原生方案與套件方案的選擇理由。
3. 桌機 1440 px、平板 768 px、手機 360 px 的畫面紀錄。
4. 鍵盤、觸控、reduced-motion 與失敗 fallback 的測試結果。
5. 預估工時與實際工時差異。
6. 所需素材清單與素材製作時間。
7. 初始 JS、第三方請求、主要媒體大小與基本效能紀錄。
8. 「採用／條件採用／保留研究／不採用」結論。
9. 若採用，整理成可重用元件；若不採用，保留原因而非直接刪除研究結論。

## 12. 使用者體驗與品質底線

- 所有核心內容在動畫失敗時仍可閱讀。
- 所有聲音必須由使用者主動啟動，並能隨時關閉。
- 重要資訊不能只藏在 Carousel、hover 或桌寵對話中。
- 觸控裝置不能依賴滑鼠 hover。
- 所有動畫尊重 `prefers-reduced-motion`。
- Gallery、Carousel、Lightbox、播放器與導覽必須能以鍵盤操作。
- 第三方內容載入失敗時保留標題、來源與原文連結。
- 不用「動畫很多」作為品質指標；每個動畫都必須有敘事、導引或回饋目的。

## 13. 效能測試基準

實驗階段先採以下目標，正式站再以真實素材重新驗證：

- 首屏不直接載入多個 X、Threads 或 YouTube iframe。
- 非首屏圖片使用 lazy loading 並預留尺寸。
- 動畫主要使用 `transform` 與 `opacity`。
- 持續動畫在頁面不可見時暫停。
- 桌機動畫目標 60 FPS；一般手機不可出現持續明顯卡頓。
- 每加入一個 runtime 或第三方 script，都記錄增加的傳輸量與請求。
- Lighthouse Performance、Accessibility、Best Practices、SEO 以 90 分作為方向，不以單次本機分數作唯一判斷。

## 15. 初步風險清單

| 風險 | 可能影響 | 對策 |
|---|---|---|
| 實驗功能過多 | 永遠停留在研究，正式網站無法推進 | 每階段設停止點，只挑選對正式站有價值的項目 |
| 把 Prototype 當正式版 | 上線後出現 RWD、無障礙與錯誤處理問題 | Prototype 與正式整合工時分開 |
| 素材時間被忽略 | Live2D、小遊戲與動畫嚴重延誤 | 美術、建模、音效與程式分開估時 |
| 提前安裝大量套件 | 相容性、升級與 bundle 成本增加 | 原生方案先行，逐項取得同意 |
| 第三方平台改版 | X／Threads embed 失效或樣式改變 | 延遲載入、fallback、原文連結與定期抽查 |
| 動畫過度 | 暈動、手機掉幀、內容難讀 | 動態偏好、效能模式、手機降級 |
| 實驗區意外公開 | 未授權素材或半成品被搜尋 | `noindex`、排除 Sitemap，正式部署前再確認 |
| Live2D 授權判斷錯誤 | 無法發布或產生費用 | 在使用 SDK 前依發布主體與用途重新核對官方授權 |

## 16. 里程碑

### L1　實驗室可用

- `/lab/` 索引、共用樣式與結果模板建立。
- 不影響正式首頁，production build 成功。

### L2　原生能力基線完成

- Navigation、Transition、Scroll、Gallery、Carousel、Lightbox、影音與基本動效完成比較。
- 得到「不用額外套件時」的品質與工時基準。

### L3　外部平台整合完成

- X、Threads、YouTube 皆有延遲載入與 fallback。
- 確認第三方內容的隱私、效能與維護成本。

### L4　進階功能決策完成

- 桌寵、小遊戲、人物動畫與 Islands 至少選兩項實驗。
- 決定正式網站採用的動畫層級與互動上限。

### L5　活動估時模型完成

- 所有已測功能都有預估／實際工時。
- 形成簡單版、標準版、進階版三種活動頁套餐與合理工期。

## 17. 建議的第一個實驗

第一個實驗建議為「Navigation ＋ Page Transition 比較」，原因如下：

- 可直接回答單頁錨點、多頁刷新與類 SPA 體驗的差異。
- Astro 內建能力即可，不需新 dependency。
- 能先確立後續每個 `/lab/` 實驗頁的導覽方式。
- 可測試原生 MPA、瀏覽器 View Transition 與 `<ClientRouter />` 三種模式。
- 風險較低，且能建立後續第三方 script 生命週期的測試基礎。

本企劃通過後，仍應先只建立 Phase 0 與第一個實驗，不一次展開全部功能。

在開始 Phase 0 前，需要使用者逐步確認：

4. 是否接受每次只做一項、驗收後再進下一項。
